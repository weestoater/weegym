# GitHub Copilot Instructions for WeeGym

**Priority Order:** When rules conflict, apply them in this order: (1) Security, (2) Testing, (3) Code Quality. For example, never use console.log even in test files under src/.

## Commit Message Requirements

**ALWAYS use Conventional Commits format for all commit messages.**

### Required Format

```
type(scope): subject

body (optional)

footer (optional)
```

### Commit Types

- `feat:` - New feature (triggers minor version bump)
- `fix:` - Bug fix (triggers patch version bump)
- `refactor:` - Code refactoring that changes behavior or structure (triggers patch version bump)
- `perf:` - Performance improvement (triggers patch version bump)
- `docs:` - Documentation only (no release)
- `test:` - Test changes (no release)
- `chore:` - Maintenance tasks (no release)
- `style:` - Code formatting only (whitespace, semicolons, indentation) with NO functional changes (no release)
- `ci:` - CI/CD changes (no release)

**Note:** Use `style:` only for pure formatting changes (e.g., fixing indentation, adding/removing whitespace). Use `refactor:` when renaming variables, reorganizing code structure, or making semantic changes even if behavior is preserved.

### Examples

```bash
feat: add Strava webhook support
fix: resolve activity sync duplication
refactor(auth): simplify login flow
docs: update API documentation
test: add calorie estimator tests
```

### Rules (by priority)

1. **(Required)** Type must be from the approved list above
2. **(Required)** Subject must be lowercase and under 50 characters
3. **(Required)** Use present tense ("add" not "added")
4. **(Required)** No period at end of subject
5. **(Preferred)** Include scope for module-specific changes (e.g., `feat(auth):`, `fix(strava):`)
6. **(Preferred)** Be specific and descriptive

### Breaking Changes

For breaking changes, add `!` after type:

```bash
feat!: redesign settings API

BREAKING CHANGE: Settings now use nested structure
```

## Code Practices

### Security

- Never commit console.log statements in src/
- No credentials or secrets in code
- Security checks run on all commits

### Testing

- Add tests for new features
- Maintain line coverage above 70% as reported by `yarn test --coverage`. Branch coverage should also stay above 60%
- Run `yarn test` before committing

### Code Quality

- Follow ESLint rules
- Use descriptive variable names
- Add comments for complex logic
- Keep functions small and focused

## Project-Specific Conventions

### Strava Integration

- All Strava API calls go through `src/services/stravaService.js`
- Use incremental sync by default
- Always handle rate limiting
- **Error Handling:** On rate limit (HTTP 429), implement exponential backoff starting at 1 second, max 3 retries, using the `retryWithBackoff` utility in `src/utils/retry.js`. Log rate limit events with `console.warn` only outside of `src/`

### Supabase

- All database queries use Supabase client from `src/lib/supabase.js`
- Enable RLS policies for all tables
- Use typed queries where possible

### React Components

- Functional components with hooks
- PropTypes for all components
- Keep components focused and reusable

## When Suggesting Commits

Always suggest commit messages in conventional commit format. If the user provides a commit message that does not follow Conventional Commits format, respond by:

1. Identifying the specific formatting issue (e.g., missing type, incorrect capitalization, no scope when needed)
2. Providing a corrected version prefixed with "Consider using:"

Example:

```
User: "git commit -m 'fixed the bug'"
You: "The message is missing a commit type prefix and uses past tense. Consider using: git commit -m 'fix: resolve activity sync bug'"
```
